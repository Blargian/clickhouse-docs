# OLAPとは何ですか？ {#what-is-olap}

[OLAP](https://en.wikipedia.org/wiki/Online_analytical_processing)はオンライン分析処理を指します。これは広い意味を持つ用語で、技術的な視点とビジネス的な視点の2つの観点から見ることができます。しかし、非常に高いレベルで見れば、これらの言葉を逆に読むこともできます：

処理
:   何らかのソースデータが処理されます...

分析
:   ...いくつかの分析レポートやインサイトを生成します...

オンライン
:   ...リアルタイムで。

## ビジネスの視点から見たOLAP {#olap-from-the-business-perspective}

近年、ビジネス関係者はデータの価値を認識し始めています。盲目的に判断する企業は、競争に追いつけなくなることが多いです。成功した企業のデータ駆動型アプローチは、ビジネス判断を下すためにリモートで有用なすべてのデータを収集する必要があり、それらをタイムリーに分析するためのメカニズムが必要です。ここにOLAPデータベース管理システム（DBMS）が登場します。

ビジネスの観点から言えば、OLAPは企業が継続的に運用活動を計画、分析、報告することを可能にし、効率を最大化し、コストを削減し、最終的には市場シェアを獲得することができます。これは、社内システムで実施することも、ウェブ/モバイル分析サービスやCRMサービスなどのSaaSプロバイダーに外部委託することも可能です。OLAPは多くのBIアプリケーション（ビジネスインテリジェンス）の背後にある技術です。

ClickHouseはOLAPデータベース管理システムで、特定のドメインデータを分析するためのこれらのSaaSソリューションのバックエンドとしてよく使用されます。しかし、一部の企業は依然として第三者プロバイダーとデータを共有することに消極的であり、社内データウェアハウスシナリオも実現可能です。

## 技術の視点から見たOLAP {#olap-from-the-technical-perspective}

すべてのデータベース管理システムは、OLAP（オンライン **分析** 処理）とOLTP（オンライン **トランザクション** 処理）の2つのグループに分類できます。前者は、大量の履歴データに基づくレポートを構築することに焦点を当てていますが、それをあまり頻繁には行いません。一方、後者は通常、継続的なトランザクションのストリームを処理し、データの現在の状態を常に変更します。

実際には、OLAPとOLTPはカテゴリーではなく、むしろスペクトルのようなものです。ほとんどの実際のシステムはそのどちらかに焦点を当てていますが、もう一方の種類のワークロードが求められる場合にはいくつかのソリューションや回避策を提供します。この状況は、企業に複数のストレージシステムを統合して運用することを強いることが多く、大した問題ではないかもしれませんが、システムが多くなると維持コストが高くなる可能性があります。したがって、近年のトレンドはHTAP（**ハイブリッドトランザクショナル/分析処理**）であり、両方の種類のワークロードを単一のデータベース管理システムで等しくうまく処理できるようにすることです。

DBMSが純粋なOLAPまたはOLTPから始まったとしても、競争に追いつくためにはHTAPの方向に進むことを余儀なくされています。そしてClickHouseも例外ではなく、最初は[できるだけ速いOLAPシステム](../../concepts/why-clickhouse-is-so-fast.md)として設計されており、今でも完全なトランザクションサポートはありませんが、データの更新や削除のための一貫した読み書きや変異のような機能は追加される必要がありました。

OLAPとOLTPシステム間の基本的なトレードオフは次のとおりです：

- 効率的に分析レポートを構築するためには、カラムを別々に読み取れることが重要であるため、ほとんどのOLAPデータベースは[列指向](../../faq/general/columnar-database.md)です。
- 一方、カラムを別々に保存することは、追加やインプレース修正といった行に対する操作コストを、カラムの数に比例して増加させます（システムがイベントの詳細をすべて収集しようとする場合、それは非常に多くなる可能性があります）。したがって、ほとんどのOLTPシステムは行ごとにデータを格納します。
